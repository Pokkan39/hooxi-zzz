# HOOXI PLAY 网站搭建计划书

## 1. 项目目标

HOOXI PLAY 的目标不是普通资料目录，而是以“凌晨录像店”为叙事空间的《绝区零》剧情档案站。访客从店外进入，通过开门、进店、自由翻架或咨询接待邦布完成内容导航，形成连续、电影化且可操作的浏览体验。

成功标准：

- 首次访问具备“店外待机 → 用户开门 → 进入店内”的清晰仪式感。
- 店内以左侧自助书目和右侧邦布接待形成两种导航方式。
- 页面滚动时场景、文案和内容模块连续衔接，不像独立卡片拼接。
- 桌面端有高质感空间动画，移动端有明确降级且保持完整内容能力。
- 编辑人员不改代码也能维护文字、图片、链接、角色与阵营资料。
- AI 只检索本站已发布内容，不臆造剧情，不在浏览器暴露服务密钥。

## 2. 当前交付基线

当前 `tape-wall-sample.html` 是可运行的 2.5D 原型：

- CSS/DOM 构建 HOOXI PLAY 店外、门头、橱窗、双开门和镜头推进。
- 店内提供书目分类、VHS 磁带墙、CRT 看片台和邦布接待台。
- 邦布接待台当前使用站内关键词导航，并明确标记 DeepSeek 待接入。
- 支持鼠标、触摸、键盘、减少动态效果偏好和移动端布局。
- 原型与正式 `index.html` 隔离，确认视觉方向后再决定是否替换首页。

这套实现适合验证信息架构和交互节奏，但不是真正的三维空间。

## 3. 图片与 3D 模型的选择

### 3.1 只有图片时

单张店铺图片可用于 2.5D：把招牌、门、玻璃、前景遮挡物、背景和灯光拆成透明图层，再使用视差、景深、遮罩和镜头缩放制造空间感。

适用范围：

- 固定机位的首屏。
- 小幅左右视差和推近。
- 门体遮罩转场。
- 较低成本的移动端版本。

限制：不能可靠改变机位、绕过物体或连续展示店外到店内的真实空间，过度推近会暴露图层边缘和透视错误。

### 3.2 有 3D 建模文件时

要实现高端定制站的连续镜头，建议提供 `.glb` 或 `.gltf`：

- 模型包含店外立面、门、橱窗、室内前台、书架、地面和主要装饰。
- 贴图使用 PBR：Base Color、Normal、Roughness、Metallic、Emissive、AO。
- 门、灯牌、邦布和可交互物体使用清晰英文节点名。
- 材质数量尽量控制在 20 个以内，合并重复材质和静态小物件。
- 桌面首屏模型建议压缩后不超过 12 MB；移动端首屏建议不超过 5 MB。
- 纹理优先 WebP/KTX2；大面积主材质最高 2048，普通物件 1024 或更低。
- 动画可内置 Door_Open、Sign_On、Bangboo_Idle，或由页面运行时驱动节点。

如果原始文件是 `.blend`、`.fbx` 或 `.obj`，应先在 Blender 中清理、烘焙并导出 `.glb`，网页不直接加载工程源文件。

## 4. 推荐技术架构

### 4.1 展示层

- 保持原生 HTML/CSS/JavaScript 作为内容和无障碍基线。
- 3D 阶段使用 Three.js 加载 GLB；只为 3D 场景引入该依赖，不重写全部页面。
- 使用 `requestAnimationFrame` 驱动镜头，滚动进度由原生 ScrollTimeline（可用时）或轻量滚动映射控制。
- 文案、按钮和表单继续使用真实 DOM，避免把可访问内容画进 Canvas。
- 使用 IntersectionObserver 延迟加载店内模型、角色卡图和非首屏视频资源。

### 4.2 内容层

- `data.js` 继续保存页面、剧情、角色、阵营与自由组件内容。
- `layout-data.js` 只保存桌面、平板、手机的布局偏移和层级。
- 图片和 3D 文件放入 `assets/`，所有发布内容使用仓库相对路径。
- 编辑器通过稳定 ID 绑定内容，不依赖 DOM 顺序。

### 4.3 AI 服务层

DeepSeek 不能从静态页面直接调用，因为浏览器会暴露 API Key。正式接入采用：

1. 浏览器向同源后端发送用户问题。
2. 后端校验长度、频率、来源和会话。
3. 后端从本站结构化数据构建检索结果，只把相关条目发给 DeepSeek。
4. DeepSeek 生成带站内链接的简短回答。
5. 后端过滤未知链接与无来源结论后返回前端。

建议接口：

```text
POST /api/concierge/search
{ "query": "我想看安比的个人故事" }
```

服务端环境变量使用 `DEEPSEEK_API_KEY`，不得写入 `data.js`、前端脚本或 Git 历史。第一阶段无需向量数据库，先对标题、摘要、角色、阵营、版本和标签做确定性检索；内容规模增长后再评估嵌入索引。

## 5. 多段滚动叙事

建议把首页拆成六个连续场景：

1. **夜街待机**：固定远景，灯牌熄灭，显示点击或 Enter 提示。
2. **店门开启**：灯牌点亮，门体打开，镜头沿门洞推进。
3. **前台欢迎**：邦布进入待机动画，说明两种浏览路线。
4. **左侧书架**：镜头沿书架横移，剧情、角色、活动、幕后分类依次出现。
5. **看片台**：抽出的磁带飞向 CRT，显示档案摘要和入口。
6. **接待终端**：邦布问答与全站导航，最后连接页脚和正式内容页。

每个场景使用独立滚动区间，镜头只在该区间插值。滚动过快时直接跳到对应状态，不排队播放所有动画。返回页面时恢复到可阅读状态，不强制重复开门。

## 6. 视觉与动效规范

- 主色：HOOXI 黄、CRT 青、深夜蓝黑、木质棕和少量警示红。
- 门头使用原创 HOOXI PLAY 字标，不直接复制原作 Logo 文件。
- 动效优先改变 transform 与 opacity，避免频繁触发布局。
- 交互反馈控制在 120–260ms；开门主转场约 1.8–2.5s。
- 所有强动画提供 `prefers-reduced-motion` 降级。
- 页面不可因动画锁死滚动；用户随时可跳过进入内容。
- 音效默认关闭，只在用户明确操作后播放，并提供静音控制。

## 7. 性能预算

桌面端目标：

- LCP 小于 2.5 秒。
- 首屏 JavaScript 压缩后小于 250 KB（不含按需加载的 Three.js 与模型）。
- 首屏总传输量在 4G 环境下小于 3 MB；3D 资源点击或空闲后加载。
- 动画主线程长任务小于 50ms，目标 50–60 FPS。

移动端目标：

- 默认使用 2.5D 店外或低模版本。
- 低性能、节流或省流量设备不加载高精度模型。
- 图片使用响应式尺寸和懒加载。
- 不依赖 hover，触控目标至少 44×44px。

## 8. 无障碍与容错

- 开门按钮必须是真实按钮，支持 Enter 和 Space。
- 提供跳过动画并直接进店的链接。
- Canvas 仅作装饰，所有导航都有 DOM 等价入口。
- 键盘焦点清晰，抽取磁带和邦布建议均可键盘操作。
- JavaScript 或 3D 加载失败时，直接显示静态书目导航。
- AI 不可用时保留当前关键词导航，不阻断站点浏览。

## 9. 分阶段实施

### 阶段 A：2.5D 交互定稿

- 评审当前 HOOXI PLAY 店外、门头、开门节奏和店内分流。
- 使用用户提供的授权店铺图片时，拆分图层并替换纯 CSS 场景。
- 完成角色/阵营/自由组件编辑覆盖。

### 阶段 B：3D 资产接入

- 接收并检查 GLB、贴图、节点与动画。
- 制作店外到前台的相机轨迹。
- 加入灯牌、门、玻璃反射和邦布待机动画。
- 建立桌面高模、移动低模和静态图三级降级。

### 阶段 C：滚动叙事与正式首页

- 将六个场景绑定滚动区间。
- 接入正式首页内容与数据编辑器。
- 完成性能分析、浏览器兼容和视觉回归后替换 `index.html`。

### 阶段 D：DeepSeek 接待

- 实现服务端同源代理、限流和站内检索。
- 增加来源链接、拒答规则和日志脱敏。
- 完成安全测试后开放真实问答。

## 10. 验收清单

- 店外、开门、进店、书架、看片台、邦布六段体验连续。
- 桌面与 390px 移动端无页面级横向溢出。
- 减少动态效果模式可直接阅读内容。
- 首屏和店内资源无 404，控制台无错误。
- 角色、阵营和自由组件可点选编辑并正确导出。
- 3D 加载失败、AI 不可用和 JavaScript 关闭时均有可用降级。
- DeepSeek Key 只存在服务端环境变量，不进入前端或仓库。
- 上线前通过性能、无障碍、内容来源和第三方素材授权复核。
